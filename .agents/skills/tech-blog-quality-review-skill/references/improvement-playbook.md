# Improvement Playbook

## A. 結論が遅い/弱い
- 冒頭 5-10 行に「誰の何をどう解決するか」を置く
- 先に最終到達点を提示し、本文はそこへ至る根拠にする

## B. 読者前提が不一致
- 冒頭で「対象」「非対象」を宣言
- 前提知識を3項目以内で明示
- 難語は初出時に1行定義を入れる

## C. 技術精度が不安
- 公式ドキュメントを一次参照として追加
- バージョン、OS、実行環境を明記
- 非推奨/落とし穴/制約を「注意」枠で分離

## D. 再現できない
- 手順を最小再現ケースに圧縮
- 各ステップに「入力」「コマンド」「期待結果」を追加
- 失敗時の代表エラーと対処を追記

## E. 長くて読了されない
- 1見出し1メッセージに分割
- 重複説明を削除し、図解/表/箇条書きへ変換
- 章末に3行サマリーを置く

## F. タイトルと導入が弱い
- タイトルは「対象 + 課題 + 得られる成果」を含める
- 導入で痛みを言語化し、記事の価値を即提示

## G. コードの説明不足
- 何を解決するコードかを先に書く
- 重要行には理由を添える（What ではなく Why）
- 実行後の確認方法と失敗条件を示す
